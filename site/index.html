Every AI coding tool remembers something different. Nothing remembers the project.

sitrep is the continuity, cost, and nudge layer for agentic AI coding — works with Claude Code, Codex, Cursor, or any tool that reads a repo.

It sits beside Jira, OpenSpec, or Spec Kit and reads whatever plan you already keep — never a replacement, never another system to maintain in parallel.

01 / Situation — why this exists

Built as a side effect, not the plan.

sitrep came out of building another AI-assisted product solo — switching between Claude Code, Codex, and Cursor depending on the day, and losing the thread every time. Which phase were we actually in? What had this feature already cost? Which tool made a decision last week, and why did it decide that?

None of the coding tools tracked any of it. They're good at writing code — none of them are the operations layer underneath the code. So that layer got built directly into the repo instead, as plain markdown a human and an AI could both read, dogfooded on itself from commit one: this repo's own STATUS_REPORT.md is sitrep, tracking sitrep.

This isn't a claim that AI coding tools have no memory at all — several now offer session resumption, custom-instruction files, or their own context tricks. What none of them do is track project state, or carry it cleanly when you switch tools on the same repo.

Multiple agentic tools now routinely touch the same codebase over its life. There's no shared layer underneath them that remembers where things stand, independent of which one is driving this week — that's the gap sitrep fills.

02 / Response — how it works

Plain markdown. One JSON file. Git as the database.

Whichever AI tool opens the repo reads exactly where the project stands at session start, and writes back automatically at session end — cost, progress, and decisions included. Nothing else to run, no account to create.

sitrep/ ├── MANIFEST.md framework version + rules ├── PROJECT_PLAN.md what to build ├── STATUS_REPORT.md where you are ├── .sitrep-data.json cost + token + user data ├── HANDOFF.md context package ├── dashboard.html visual MIS report └── history/ ├── sessions/ ├── handoffs/ └── dashboards/
03 / Signal — intelligent nudges

Not a dashboard to babysit.

sitrep watches a handful of real, observable signals — not guesses — and surfaces at most one suggestion at a time, never repeating one you've already seen. No config. Bound to your tool's mid-session hook via the same getsitrep init wizard that sets up everything else.

Each is tied to something verifiable: selfheal's own drift detection, a real count of sessions since the dashboard was last regenerated, git status showing uncommitted work.

Open Source — built to be forked, not just used

The same rules this repo holds itself to.

sitrep is dogfooded on itself — the governance below is what this repo's own history was built with, not a policy written after the fact.

Zero dependencies
The CLI uses Node.js builtins only — fs, path, readline. Adding a dependency needs a strong reason.
No build step
Every command is a single .js/.md file. No bundler, no transpiler, no compile step.
Every cost figure is labeled
No PR introduces a cost/token number without an actual/estimate/mixed label — enforced, not just documented.
Fail-open, always
Anything hook-fired must never block a developer's workflow — Hard Law #5, checked on every PR.

Good first step: open an issue, or check what's already tracked. Templates exist for bugs and features, and every PR runs against the same test suite this repo uses on itself.
